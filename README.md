RESTFull API для всіх об’єктів БД
Для всіх ORM-сутностей (Clients, DocumentTypes, SignedDocuments, Files) реалізовано REST-контролери в просторі імен EdoSign.Lab_3.Controllers.Api. Для кожного об’єкта підтримуються стандартні операції GET/GET{id}/POST/PUT/DELETE, які повертають/приймають DTO-моделі.

Контролер з двома версіями API + 2 сторінки інтерфейсу
Створено версіонований контролер ClientsController, який одночасно підтримує API v1 і API v2:

GET /api/v1/clients – спрощений DTO (тільки Id, Name), сумісний із попереднім інтерфейсом.

GET /api/v2/clients – розширений DTO (Id, Name, Email).
На PR5-інтерфейсі реалізовано дві сторінки:

«стару» сторінку, яка працює з тим самим Rest API й очікує v1-формат,

«нову» сторінку, що використовує розширені дані v2.
Обидві сторінки працюють поверх останньої реалізації контролера та не ламають зворотну сумісність.

Інтеграційні тести для 4 СУБД
Окремий Node-проєкт з тестами (Jest/Mocha) виконує інтеграційні запити до запущеного API через axios.
Тести автоматично проганяються для чотирьох варіантів БД: InMemory, Sqlite, Postgres, SqlServer (перемикання через ASPNETCORE_ENVIRONMENT / DatabaseOptions:Provider). Для кожної БД перевіряється поведінка кількох контролерів (Clients, DocumentTypes, SignedDocuments), виявлені помилки в конфігурації міграцій та моделі були виправлені.

Автогенерація коду клієнтів під час збірки
У проєкт EdoSign.Lab-3 додано інтеграцію з NSwag.
Під час збірки (dotnet build) виконується команда nswag run nswag.json, яка:

бере опис API зі Swagger (/swagger/v1/swagger.json),

генерує C#-клієнтські класи в окрему папку GeneratedClients/ApiClient.cs.
Таким чином, усі об’єкти API мають авто-згенеровані strongly-typed клієнти, що оновлюються автоматично при зміні контракту.

docker-compose / оточення для запуску
Для спрощення розгортання та тестування дозволено використання docker-compose / віртуалізованого оточення. Проєкт підтримує запуск у відокремленому середовищі (ВМ) з автоматичним деплоєм EdoAuthServer та EdoSign.Lab-3.
